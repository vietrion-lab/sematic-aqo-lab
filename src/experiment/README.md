# Experiment Framework

Benchmarking framework to evaluate Semantic-AQO vs vanilla PostgreSQL optimizer.

## How It Works

Each experiment runs a query set in **2 modes × N iterations** (default N=20):

| Mode | AQO Setting | Purpose |
|------|-------------|---------|
| `no_aqo` | `aqo.mode = 'disabled'` | Baseline — pure PostgreSQL optimizer |
| `with_aqo` | `aqo.mode = 'learn'` | Semantic AQO active, learning from each execution |

Both modes disable parallel workers (`max_parallel_workers_per_gather = 0`) for deterministic, fair comparison.

For each iteration × query, the runner captures via `EXPLAIN (ANALYZE, FORMAT JSON)`:
- **Planning Time** (ms) — optimizer overhead
- **Execution Time** (ms) — actual query running time
- **Q-error** per plan node — `max(estimated/actual, actual/estimated)`, aggregated as geometric mean

## Directory Structure

```
experiment/
├── runner.py           # Core engine: runs queries, extracts metrics from EXPLAIN JSON
├── analyze.py          # Figure generation: reads CSVs → 3 matplotlib figures
├── config.sh           # Shared settings (iterations, DB names, GUC values)
├── lib/
│   └── bench_runner.sh # Bash wrapper: calls runner.py then analyze.py
├── tpch/
│   ├── run.sh          # Entry point: run TPC-H experiment
│   ├── queries/        # Full benchmark query files (generated by setup scripts)
│   ├── test_queries/   # 3 simple queries for quick smoke tests
│   └── results/        # Timestamped output directories
└── tpcds/
    ├── run.sh          # Entry point: run TPC-DS experiment
    ├── queries/        # Full benchmark query files (generated by setup scripts)
    ├── test_queries/   # 3 simple queries for quick smoke tests
    └── results/        # Timestamped output directories
```

## Running Experiments

### Full benchmark (via pipeline script)

```bash
# Both TPC-H and TPC-DS
./scripts/run-experiment.sh --skip-load

# Individual
./scripts/run-experiment.sh --tpch-only --skip-load
./scripts/run-experiment.sh --tpcds-only --skip-load
```

### Direct runner invocation

```bash
source experiment/config.sh

# Full TPC-H (all queries, 20 iterations)
python3 experiment/runner.py tpch experiment/tpch/queries experiment/tpch/results/run1

# Quick test (3 queries, 3 iterations)
python3 experiment/runner.py tpch experiment/tpch/test_queries /tmp/test --iterations 3

# Generate figures from results
python3 experiment/analyze.py experiment/tpch/results/run1 --title "TPC-H"
```

### Individual benchmark runners

```bash
# Uses bench_runner.sh → runner.py → analyze.py automatically
./experiment/tpch/run.sh
./experiment/tpcds/run.sh
```

## Output

Each run produces a timestamped directory under `results/`:

```
results/20260312_143057/
├── no_aqo_results.csv          # Raw data: iteration, query, exec_time, plan_time, q-error
├── with_aqo_results.csv        # Same structure for AQO mode
├── figures_TPC-H.png           # Combined 3-panel figure
├── fig1_cardinality_TPC-H.png  # Q-error over iterations
├── fig2_planning_time_TPC-H.png # Planning time comparison
└── fig3_execution_time_TPC-H.png # Execution time comparison
```

```csv

```

### Figures

3 research-quality figures per benchmark:

1. **Cardinality Estimation Accuracy** — Q-error (geometric mean) averaged across all queries per iteration. Shows how AQO learns better cardinality estimates over time.

2. **Planning Time** — Average planning time per iteration. AQO has higher planning overhead due to W2V inference + k-NN prediction. This is the expected cost.

3. **Execution Time** — Average execution time per iteration. Better cardinality estimates → better query plans → faster execution. This is the payoff.

## Configuration

Edit `config.sh` to adjust:

```bash
ITERATIONS=20           # Repetitions per mode
AQO_JOIN_THRESHOLD=0    # Min joins to activate AQO (0 = all queries)
AQO_PARALLEL_WORKERS=0  # Disabled for deterministic comparison
```

## Query Files

### Full benchmarks (`queries/`)
Generated by `scripts/databases/01-setup-tpch-1gb.sh` and `02-setup-tpcds-1gb.sh`:
- TPC-H: 22 templates × 20 random seeds = up to 440 query variants
- TPC-DS: 99 templates × 20 random seeds = up to 1980 query variants
- Validated with 60s timeout; only passing queries are kept

### Test queries (`test_queries/`)
3 hand-written queries per benchmark for quick smoke testing:
- `t01.sql` — Simple aggregation with filter
- `t02.sql` — 2-table join with subquery
- `t03.sql` — Multi-table join with date predicate

## Modifying the Framework

### Add a query

Drop any `.sql` file into `queries/` or `test_queries/`. The runner discovers all `*.sql` files automatically.

### Change iterations

```bash
# Via config.sh
export ITERATIONS=50

# Or via CLI
python3 experiment/runner.py tpch queries/ results/ --iterations 50
```

### Add a new benchmark

1. Create `experiment/newbench/` with `queries/`, `test_queries/`, `results/` dirs
2. Create `experiment/newbench/run.sh` (copy from `tpch/run.sh`, change DB name)
3. Load the benchmark database
4. Run: `./experiment/newbench/run.sh`
